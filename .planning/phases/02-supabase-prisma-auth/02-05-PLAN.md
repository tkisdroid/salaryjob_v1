<anti_patterns>
- **NEVER** run seed in production. Seed script MUST throw immediately if `process.env.NODE_ENV === 'production'`.
- **NEVER** use `supabase.auth.signUp` in seed — use `supabase.auth.admin.createUser` (bypasses email confirmation, RLS, rate limits) per Pitfall #10.
- **NEVER** use `prisma.user.create` for users — the trigger creates them. Use `prisma.user.update` to override role after trigger fires.
- **NEVER** write `location` column via Prisma (Unsupported type) — use `prisma.$executeRaw` with tagged template.
- **NEVER** delete in forward FK order — always reverse: Review → Application → Job → BusinessProfile → WorkerProfile → User. Otherwise FK violations.
- **NEVER** leak service role key into output. Do NOT `console.log(process.env.SUPABASE_SERVICE_ROLE_KEY)` under any circumstances.
- **NEVER** use `new PrismaClient()` without the adapter — Supabase needs `@prisma/adapter-pg` per STACK.md.
- **NEVER** delete or modify `src/lib/mock-data.ts` — Phase 5 exit criterion.
- **NEVER** skip the `auth.admin.listUsers()` cleanup loop — without it, re-running seed leaves orphaned auth.users rows.
- **NEVER** use `prisma.auth_users.deleteMany()` — auth.users is not a Prisma model, must use Supabase admin API.
- **NEVER** hardcode project URL in seed — read from env.
- **NEVER** insert with `prisma.review.create` — reviews are empty in Phase 2 seed (Phase 5 scope).
- **NEVER** forget to `prisma.$disconnect()` in finally block — seed will hang CI.
</anti_patterns>
