# Plan 02-07 — Mock-data UI Swap (Gap Closure)

## Why this plan exists

CLAUDE.md hard constraint:
> Mock removal: Phase 2 종료 시점에 `src/lib/mock-data.ts` 의존 경로 0개여야 함. 이 파일이 남아있으면 Phase 2는 미완료.

After plans 02-01 ~ 02-06 finished, **11 files in `src/app/`** still import from `@/lib/mock-data`. This plan removes those imports without touching visual design.

## Strategy

**Thin adapter layer** — minimize changes to consuming files. Each consumer keeps its existing type names and call signatures; we redirect imports to new modules backed by Prisma.

1. **Pure utilities move** (no DB) → `src/lib/job-utils.ts`:
   - `calculateEarnings(job)`, `formatWorkDate(iso)`, `categoryLabel(cat)`, `categoryEmoji(cat)`
2. **Shared types move** → `src/lib/types/job.ts`:
   - `JobCategory`, `MockJob`, `MockApplication`, `MockWorker`, `MockBusiness`, `MockReview`, `MockBizApplicant`
   - Rename `Mock*` → `Job`, `Application`, `Worker`, `Business`, `Review`, `BizApplicant` (cleaner) — but keep old names as type aliases for backward compat
3. **Prisma-backed adapters** → `src/lib/db/queries.ts`:
   - `getJobs()`, `getJobById(id)`, `getJobsByCategory(cat)`, `getUrgentJobs()`, `getTodayJobs()`
   - `getApplications()`, `getApplicationById(id)`
   - `getCurrentWorker()` — reads from `dal.verifySession()` + `prisma.workerProfile.findUnique`
   - `getReviews()`, `getBizApplicantById(id, applicantId)`, `getBusinessById(id)`
   - Each returns objects whose **shape matches** the existing `Mock*` interfaces (so consumer code stays unchanged)
   - Use `prisma` from `src/lib/db/index.ts` (already exists from 02-05)
4. **Sweep update** all 11 files to import from new locations
5. **Verify**: `grep -r "@/lib/mock-data" src/app/` → 0 results. `npx tsc --noEmit` → 0 errors. `npx vitest run` → green.

## Critical constraints

- **DO NOT delete or modify `src/lib/mock-data.ts`** — `prisma/seed.ts` still imports from it (Plan 02-05 D-04)
- **DO NOT redesign visuals** — this is import-redirect work only
- **Page files must remain async server components where they already are** — `getJobs()` etc are async Prisma calls
- **Client components** (`apply-confirm-flow.tsx`, `check-in-flow.tsx`) take props from server parents — adapters should be called in the parent server component and passed down, not from the client component itself
- **Type rename safety**: if rename `Mock*` → bare names, old names must still work via `export type MockJob = Job` aliases for any external test/seed usage

### Task 2: Build Prisma-backed adapters

Create `src/lib/db/queries.ts`. Each function returns objects matching the existing `Mock*` shapes — convert Prisma's snake_case + DB types into the camelCase shapes that consumer code expects.

Key adapters needed:
- `async getJobs(opts?: { limit?: number; category?: JobCategory }): Promise<Job[]>`
- `async getJobById(id: string): Promise<Job | null>`
- `async getJobsByCategory(category: JobCategory): Promise<Job[]>`
- `async getUrgentJobs(): Promise<Job[]>` — filter `is_urgent = true`
- `async getTodayJobs(): Promise<Job[]>` — filter `start_date::date = current_date`
- `async getApplications(): Promise<Application[]>`
- `async getApplicationById(id: string): Promise<Application | null>`
- `async getCurrentWorker(): Promise<Worker | null>` — reads session via DAL
- `async getReviews(): Promise<Review[]>`
- `async getBizApplicantById(jobId: string, applicantId: string): Promise<BizApplicant | null>`
- `async getBusinessById(id: string): Promise<Business | null>`

For PostGIS location fields, use `$queryRaw` with `ST_X(location::geometry) AS lng, ST_Y(location::geometry) AS lat` and reconstruct the `coordinates: { lat, lng }` object.

**Commit message:** `feat(02-07): Task 2 — Prisma-backed query adapters mimicking mock-data shapes`
