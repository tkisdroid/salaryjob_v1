## Summary

Phase 07.1 assembles **four well-established QA primitives** (Supabase CLI local stack, Playwright multi-project, @axe-core/playwright, Lighthouse CI) into a single **`npm run review`** composition, wraps them with a zero-error aggregation gate, and adds a **scoped auto-fix loop** that iterates against the aggregated failure list. The individual building blocks are battle-tested; the integration risk lives in three places: (1) Supabase CLI boot reliability on Windows+WSL2+Docker Desktop (D-03 — known port-reservation issues), (2) Playwright's `networkidle` + React 19 Server-Component hydration timing under D-11's 15s budget, and (3) the auto-fix loop's ability to respect D-19/D-20 write scoping without human intervention across 3–10 iterations.

The existing project infra — `playwright.config.ts` (Chromium desktop, serial workers), `vitest.config.ts` (node + jsdom split), 19 existing e2e specs in `tests/e2e/`, `scripts/apply-supabase-migrations.ts`, and 25 supabase migrations already on disk — **provides ~60% of the substrate**. The phase adds the *review* layer as a non-overlapping sibling (`tests/review/*`, `scripts/review/*`, `docs/review-harness.md`, new `npm run review:*` scripts) so regression risk on the existing test suite is minimal.

**Primary recommendation:** Use `@lhci/cli autorun` (one-shot, no LHCI server — per Claude's Discretion in D-17), run Playwright with a **setup project** (project-dependencies pattern) that authenticates once per persona and shares `storageState.json` across both Desktop and Mobile 375px projects, filter axe-core violations post-scan by `impact === 'critical' || 'serious'` (no built-in severity filter exists — common pitfall), and treat D-11's `networkidle` assertion as a **behavior anchor** (wait for a hydration-complete testid), not a pure network quiescence check.

### Pattern 1: Supabase CLI Local Stack Bootstrap

**What:** Shell out to `supabase start` (Docker backend), wait for `supabase status` to report all services green, apply migrations, run seed script.

**When to use:** Every `npm run review` invocation. Subsequent runs can reuse the stack (`supabase status` returns fast when up).

**Example:**
```typescript
// scripts/review/start-local-stack.ts
// Source: https://supabase.com/docs/guides/local-development/cli/getting-started (HIGH)
import { spawnSync } from 'node:child_process';
import { writeFileSync } from 'node:fs';

export async function startLocalStack(): Promise<{ apiUrl: string; dbUrl: string }> {
  // 1. Check Docker is reachable before `supabase start` (fail fast with actionable msg per D-03)
  const dockerInfo = spawnSync('docker', ['info'], { encoding: 'utf8' });
  if (dockerInfo.status !== 0) {
    throw new Error('Docker Desktop not reachable. On Windows, open Docker Desktop and wait for it to show "Running". See docs/review-harness.md.');
  }

  // 2. Start stack (idempotent; returns fast if already running)
  const start = spawnSync('npx', ['supabase', 'start'], { stdio: 'inherit', shell: true });
  if (start.status !== 0) throw new Error('supabase start failed — see stderr above.');

  // 3. Health check via `supabase status --output env` (machine-parseable)
  const status = spawnSync('npx', ['supabase', 'status', '--output', 'env'], { encoding: 'utf8', shell: true });
  if (status.status !== 0) throw new Error('supabase status failed.');

  // 4. Write .env.test (D-04 defaults) for Playwright/Prisma to consume
  writeFileSync('.env.test', status.stdout, { mode: 0o600 });
  return {
    apiUrl: 'http://127.0.0.1:54321',
    dbUrl: 'postgresql://postgres:postgres@127.0.0.1:54322/postgres',
  };
}
```

### Pattern 3: Per-Route 5-Assertion Helper (D-11)

**What:** Reusable helper returning a structured assertion-result object so the aggregator can report which sub-assertions failed (D-11 closing note: "each assertion is independently FAIL-capable").

**When to use:** Every route-scenario (108 total = 54 × 2 viewports).

**Example:**
```typescript
// tests/review/helpers/page-ready.ts
// Source: D-11 in CONTEXT.md; Playwright networkidle caveat — https://playwright.dev/docs/api/class-page#page-wait-for-load-state (HIGH)
import type { Page } from '@playwright/test';

export type PageReadyResult = {
  a_status: boolean; a_status_msg?: string;
  b_loaded: boolean; b_loaded_msg?: string;
  c_content: boolean; c_content_msg?: string;
  d_textLen: boolean; d_textLen_msg?: string;
  e_noSkeleton: boolean; e_noSkeleton_msg?: string;
};

export async function checkPageReady(
  page: Page,
  path: string,
  contentAssertion: string | RegExp,
): Promise<PageReadyResult> {
  const r: PageReadyResult = { a_status: false, b_loaded: false, c_content: false, d_textLen: false, e_noSkeleton: false };

  // (a) status 200 + no redirect loop (track via response chain)
  const response = await page.goto(path, { waitUntil: 'commit' });
  if (!response) { r.a_status_msg = 'no response'; return r; }
  r.a_status = response.status() >= 200 && response.status() < 400;

  // (b) domcontentloaded + networkidle within 15s
  try {
    await page.waitForLoadState('domcontentloaded', { timeout: 5_000 });
    await page.waitForLoadState('networkidle',       { timeout: 15_000 });
    r.b_loaded = true;
  } catch (e) { r.b_loaded_msg = (e as Error).message; }

  // (c) content assertion — either [data-testid="page-root"] OR manifest-specified anchor
  const root = page.locator('[data-testid="page-root"]');
  if (await root.count() > 0) r.c_content = true;
  else {
    const anchor = typeof contentAssertion === 'string'
      ? page.getByText(contentAssertion, { exact: false })
      : page.getByText(contentAssertion);
    if (await anchor.count() > 0) r.c_content = true;
    else r.c_content_msg = 'neither page-root testid nor anchor text found';
  }

  // (d) body text > 50 chars
  const txt = (await page.locator('body').innerText()).trim();
  r.d_textLen = txt.length > 50;
  if (!r.d_textLen) r.d_textLen_msg = `only ${txt.length} chars visible`;

  // (e) no loading skeleton/spinner after 5s stabilization
  await page.waitForTimeout(5_000);
  // Selector list MUST live in tests/review/helpers/loading-states.ts (D-11 apply)
  const { LOADING_SELECTORS } = await import('./loading-states');
  const stillLoading = await Promise.all(LOADING_SELECTORS.map((s) => page.locator(s).count()));
  r.e_noSkeleton = stillLoading.every((n) => n === 0);
  if (!r.e_noSkeleton) r.e_noSkeleton_msg = `loading element still visible: ${LOADING_SELECTORS[stillLoading.findIndex((n) => n > 0)]}`;

  return r;
}
```

**Important pattern note (HIGH confidence, from Playwright docs):** `networkidle` is officially **discouraged** for testing because it waits 500ms after network silence, which doesn't compose well with Next 16 RSC streaming. D-11b requires it regardless — **implement as specified** but add D-11c (content assertion) as the real readiness gate. This is exactly the pattern Next.js/Playwright docs recommend: "Load the page with a behaviour anchor: wait for an element that signals readiness."

### Pattern 4: A11y — Critical+Serious Filter (D-12)

**What:** `@axe-core/playwright` has **no built-in severity filter**. You must filter the `violations[]` array by `impact`.

**Example:**
```typescript

import AxeBuilder from '@axe-core/playwright';
import type { Page } from '@playwright/test';

export async function runAxeCriticalSerious(page: Page) {
  const results = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])  // standard WCAG baseline
    .analyze();

  const blocking = results.violations.filter(
    (v) => v.impact === 'critical' || v.impact === 'serious',
  );

}
```

### Pattern 6: Auto-Fix Loop with Scoped Write Authority (D-19/D-20/D-21/D-22)

**What:** A Node script that:
1. Runs full sweep, aggregates failures into a typed JSON manifest (`{ gate, route, persona, file_suggestion, message }[]`).
2. For each failure, consults D-19 whitelist and D-20 denylist to decide if it's in-scope.
3. Invokes the fix generator (Claude's Discretion per D-22 — LLM diff, regex, AST) with a **system-prompt-level deny list** (belt-and-suspenders over the filesystem-level check).
4. Writes changes only to whitelisted paths. ANY attempted write to a denied path = abort + `MANUAL-FIX-NEEDED.md` entry.
5. Re-runs sweep; compares `fail_count[iter_n]` vs `fail_count[iter_n-1]`.
6. Loops per D-21: 3 iter default, 10 if strict decrease, PAUSE after 3 no-progress.

**When to use:** Invoked by Plan 02 after the initial full sweep reports failures.

**Example skeleton:**
```typescript
// scripts/review/auto-fix-loop.ts
// Source: D-19..D-22 in CONTEXT.md; design analog: Playwright Healer (https://playwright.dev/docs/test-agents) (MEDIUM — conceptual, not verbatim API)
import { execSync } from 'node:child_process';
import { writeFileSync, appendFileSync } from 'node:fs';

const WHITELIST_PATTERNS = [
  /^src\/components\//,
  /^src\/app\/.*\/(page|layout|loading|error|not-found)\.tsx$/,
  /^src\/lib\/actions\//,
  /^src\/app\/api\/(?!webhooks\/).*\/route\.ts$/,
  /^src\/lib\/services\/(?!payment).*/,
  /^src\/lib\/validations\//,
  /^src\/styles\//,
  /^tests\//,
];
const DENYLIST_PATTERNS = [
  /^src\/lib\/supabase\//,
  /^prisma\/schema\.prisma$/,           // core-field check done in diff review
  /^supabase\/migrations\//,
  /^\.env/,
  /^next\.config\.ts$/,
  /^package\.json$/,
  /^src\/app\/api\/webhooks\//,
  /^src\/lib\/services\/payment/,
  /^middleware\.ts$/,
  // Added by CLAUDE.md policy — LEG-03 regression prevention:
  /^src\/lib\/mock-data/,
];

export function isAllowedPath(relPath: string): boolean {
  if (DENYLIST_PATTERNS.some((p) => p.test(relPath))) return false;
  return WHITELIST_PATTERNS.some((p) => p.test(relPath));
}

export async function runAutoFixLoop(): Promise<'green' | 'paused' | 'exhausted'> {
  let failures = await runFullSweep();
  let prev = failures.length;
  let noProgress = 0;
  const MAX_ITER_STRICT = 3;
  const MAX_ITER_PROGRESS = 10;
  let iter = 0;

  while (failures.length > 0) {
    iter++;
    if (iter > MAX_ITER_PROGRESS) return 'exhausted';

    // Generate diffs for in-scope failures
    const proposals = await generateFixes(failures);
    const accepted = proposals.filter((p) => isAllowedPath(p.file));
    const rejected = proposals.filter((p) => !isAllowedPath(p.file));
    for (const r of rejected) {
      appendFileSync('MANUAL-FIX-NEEDED.md',
        `- [${r.file}] ${r.reason} — blocked by D-20; ${r.gate} gate at ${r.route}\n`);
    }

    // Apply + commit per executor directive
    for (const p of accepted) writeFileSync(p.file, p.newContent);
    execSync(`git add -A && git commit -m "fix(07.1-02): iter ${iter} — ${accepted.length} fixes, ${rejected.length} escalated [iter=${iter}]"`);

    failures = await runFullSweep();
    const curr = failures.length;
    const strict = curr < prev;
    console.log(`iter=${iter} failures={${prev}→${curr}} progress=${strict ? 'yes' : 'no'}`);
    prev = curr;

    if (!strict) noProgress++;
    else noProgress = 0;

    if (iter >= MAX_ITER_STRICT && !strict) return 'paused';  // Hand off to user per D-21
    if (noProgress >= 3)                    return 'paused';
  }
  return 'green';
}
```

### Anti-Patterns to Avoid

- **Trusting `networkidle` as readiness** — waits only for 500ms silence; RSC streaming + React 19 hydration often keep requests in-flight. Use content assertion (D-11c) as the true gate.
- **One login per test** — 19 existing tests already suffer from this; with 108 route-scenarios × 3 personas = 324 logins if per-test. Use setup-project pattern (Pattern 2).
- **Short-circuit on first failure** — D-17 explicitly requires aggregation. Continue-on-failure per test; aggregate at the end.
- **Auto-fix without git commits per iteration** — loses diff-level reviewability. Executor directive already mandates `fix(07.1-02): ... [iter=N]`.
- **Running vitest in Playwright config scope** — vitest.config.ts excludes `tests/e2e/**`; do the same for `tests/review/**` to avoid double-run (current exclude covers e2e but not review).
- **Using `page.waitForNavigation()`** — deprecated in favor of `page.waitForURL()` (Playwright 1.50+).

## Common Pitfalls

### Pitfall 1: Supabase CLI Port Collisions on Windows (D-03/D-04)

**What goes wrong:** `supabase start` fails with "port is unavailable due to reserved range (WSAEACCES)" even though nothing else is listening on 54321–54324. Windows' Hyper-V Network Service (HNS) claims random TCP ranges at boot.

**Why it happens:** Windows' dynamic port range reservation overlaps with Supabase defaults. Issue #4128 in the Supabase CLI repo.

**How to avoid:**
1. Document the `net stop hns && net start hns` (admin prompt) workaround in `docs/review-harness.md`.
2. Bootstrap script should run `netsh interface ipv4 show excludedportrange protocol=tcp` on Windows and surface a specific error message if 54321–54324 fall in the reserved range.
3. Do NOT shift ports — D-04 locks defaults so test fixtures/URLs stay simple.

**Warning signs:** `supabase start` hangs at "Creating container..." or exits immediately with `bind: permission denied`.

### Pitfall 2: Docker Desktop Memory Starvation

**What goes wrong:** Supabase stack (Postgres + GoTrue + Storage + Realtime + PostgREST + Studio + Inbucket + edge-runtime) needs ~3 GB. Running alongside Chromium (Playwright + LHCI) + `next dev` can push a 4 GB WSL2 allocation into swap, causing 60+ second response times that fail D-11b's 15s timeout.

**Why it happens:** WSL2 default is 50% of host RAM (often < what we need once Chrome starts). Docker Desktop allocates inside WSL2. Swap kicks in silently.

**How to avoid:** `.wslconfig` with `memory=8GB`, `processors=4`, `autoMemoryReclaim=gradual`. Document in `docs/review-harness.md`. Bootstrap script can query `wsl --status` + `docker info --format {{.MemTotal}}` and warn if < 6 GB.

**Warning signs:** First review run is green; second run (after a few flakes) times out on `waitForLoadState('networkidle')` for all routes.

### Pitfall 3: Playwright `networkidle` Inside Next.js 16 RSC Streaming

**What goes wrong:** D-11b requires `networkidle` within 15s. Next 16 App Router + React 19 streaming RSC keeps chunks in-flight for several hundred ms after visual completion. In dev mode, HMR websocket + React Fast Refresh polling can prevent network ever going truly idle.

**Why it happens:** `networkidle` waits for **500 ms** of no network activity. HMR websocket can trigger heartbeat packets that reset that window. Playwright docs **explicitly discourage** `networkidle` for testing.

**How to avoid:**
1. Run review against a **production build** (`next build && next start`), not `next dev`. Decision candidate for planner — not in D-01..D-22, but strongly recommended. Add `REVIEW_RUN=1` causing `webServer.command` to use `next start`.
2. If dev mode is required, lower D-11b threshold impact: keep the networkidle requirement but ensure D-11c (content anchor) is the true readiness gate — FAIL only if **both** (b) times out AND (c) anchor not found.
3. Suppress HMR websocket specifically — block `ws://localhost:*/_next/webpack-hmr` via `page.route()` at review-mode startup.

**Warning signs:** Flaky FAIL on D-11b but (c)/(d)/(e) all pass. Timeouts cluster around routes with `useSWR`/TanStack Query polling (`/my/applications`, `/biz/posts/[id]/applicants`).

### Pitfall 4: axe-core Severity Filter Doesn't Exist

**What goes wrong:** Developer tries `AxeBuilder.withImpact('critical')` or `.disableMinor()` — neither exists. Tests compile, run, but report 0 violations because no filter was actually applied.

**Why it happens:** `@axe-core/playwright` exposes `withTags()` / `disableRules()` / `withRules()` / `include()` / `exclude()` — but NO severity-based filter. Severity (`impact`) is a property of the returned violation, not an input to the scanner.

**How to avoid:** Filter post-scan: `results.violations.filter((v) => v.impact === 'critical' || v.impact === 'serious')`. Shown verbatim in Pattern 4 above. Write a unit test over the filter function so a misconfigured review doesn't silently pass.

**Warning signs:** A11y gate (G8) reports 0 violations on a route with obviously missing alt text.

### Pitfall 5: LHCI Chromium Mismatch

**What goes wrong:** LHCI auto-downloads its own puppeteer Chromium (~200 MB) on first run. In WSL2 without network during `supabase start` phase, this stalls. Also, version skew between LHCI's Chromium and Playwright's Chromium can produce different LCP readings.

**Why it happens:** LHCI chrome-path auto-discovery doesn't know about Playwright's browser cache.

**How to avoid:** `chromePath: chromium.executablePath()` in `.lighthouserc.js` (Pattern 5). Pre-run sanity check: `npx playwright install chromium` before `lhci autorun`.

**Warning signs:** LHCI first run "hangs" for several minutes downloading; or LCP values differ from Playwright trace by >200ms on the same page.

### Pitfall 6: Auto-Fix Loop Breaks Denylist via "Close Enough" Paths

**What goes wrong:** Auto-fix agent edits `src/lib/services/settlement.ts` (NOT in denylist by path but D-20 says "excluding if it touches payments") or `prisma/schema.prisma` core field while claiming it only added an enum value. Review verifier catches it only after the commit.

**Why it happens:** D-20's exceptions are semantic ("if it touches payments", "core fields") not purely path-based. A filesystem check (Pattern 6's `isAllowedPath`) can't distinguish "added enum value" from "changed FK".

**How to avoid:**
1. Pre-commit hook that runs `git diff --cached -- prisma/schema.prisma` and rejects if the diff changes anything other than an `enum { ... }` block body.
2. Treat `src/lib/services/payment*`, `src/lib/services/settlement.ts`, `src/lib/services/toss*` as hard denies at filesystem level even though D-20 says "if". Safer interpretation.
3. Verifier per Success Criteria #7 runs `git log --stat phase_start..HEAD` — make the planner set `phase_start` tag explicitly at Plan 01 Task 1 start so this check is deterministic.

**Warning signs:** `MANUAL-FIX-NEEDED.md` has 0 entries but `prisma/schema.prisma` has changes.

### Pitfall 7: TESTING.md Is Stale — Don't Trust It

**What goes wrong:** `.planning/codebase/TESTING.md` claims "No testing framework configured / No test files present" — actually the project has `vitest@^3.2.4`, `@playwright/test@^1.59.1`, 19 e2e specs, and 14 test subdirectories (`tests/admin`, `tests/applications`, `tests/auth`, etc.).

**Why it happens:** Doc was generated at codebase scout time before Phase 4+. CONTEXT.md explicitly flags this (canonical refs section): "⚠ STALE (claims no tests exist; actually vitest + playwright are set up). Plan 01 Task 1 must refresh this doc."

**How to avoid:** Plan 01 Task 1 MUST include rewriting `.planning/codebase/TESTING.md` with the actual state, enumerating the existing framework versions, test subdirectories, and the new `tests/review/*` layout. This is Success Criteria #11.

**Warning signs:** Future agents planning test work read TESTING.md, conclude "we need to install vitest", and propose duplicate infra.

### Pitfall 8: Existing `tests/e2e/*.spec.ts` Accidentally Run Under Review Projects

**What goes wrong:** When adding `review-desktop` and `review-mobile-375` projects with `testDir: './tests/review'`, a misconfiguration (e.g., forgetting to set `testDir` per project) makes the review projects also run the 19 existing e2e specs — doubling runtime and potentially re-authenticating with conflicting credentials.

**Why it happens:** Playwright project `testDir` defaults to the config-level `testDir`. Without a per-project override, Playwright picks up any matching file.

**How to avoid:** Always set `testDir` explicitly on the review projects (shown in Pattern 2). Add a smoke assertion in the review runner that counts specs per project and bails if review-desktop picks up > 108 + 7 + setups.

### Pitfall 9: Vitest File Glob Picks Up `tests/review/*.spec.ts`

**What goes wrong:** `vitest.config.ts` current exclude is `['node_modules', 'tests/e2e/**', '.next']`. If new `tests/review/**` grows spec-like files, vitest will try to run them in Node and hit `test.describe()` errors (Playwright API not available outside Playwright runner). STATE.md Known Risks already mentions this for Phase 5.

**How to avoid:** Update `vitest.config.ts` exclude to add `'tests/review/**'` as part of Plan 01 Task 1. (Not in D-01..D-22 but logical consequence of `tests/review/*` sibling layout.)

```typescript
import type { Page } from '@playwright/test';

export function hookErrorCollectors(page: Page) {
  const jsErrors: Error[] = [];
  const consoleErrors: string[] = [];
  const consoleWarnings: string[] = [];
  const badResponses: { url: string; status: number }[] = [];

  page.on('pageerror', (err) => jsErrors.push(err));
  page.on('console', (msg) => {
    if (msg.type() === 'error') consoleErrors.push(msg.text());
    if (msg.type() === 'warning') consoleWarnings.push(msg.text());
  });
  page.on('response', (res) => {

  });

  return { jsErrors, consoleErrors, consoleWarnings, badResponses };
```

```typescript
import fg from 'fast-glob';

export async function discoverRoutes(): Promise<string[]> {
  const files = await fg('src/app/**/page.tsx', { cwd: process.cwd() });
  return files.map((f) => {
    // src/app/(worker)/home/page.tsx → /home
```

```typescript
export async function probeCta(page: Page, selector: string): Promise<'url' | 'modal' | 'toast' | 'network' | 'no-response'> {
```

## Open Questions

### 1. LCP ≤ 2.5s budget against `next dev` vs `next start` (G9)

- **What we know:** `next dev` ships development React with extra overhead; production `next start` is the realistic target. D-17 doesn't specify which.
- **What's unclear:** Whether the 2.5s LCP budget is achievable against `next dev` on any route. If not, review harness MUST run against `next start`.
- **Recommendation:** Default `REVIEW_RUN=1` to build + start (not dev). Add as Claude's-Discretion decision in Plan 01. If user wants `dev` mode, relax G9 by 500ms explicitly — but that's a D-17 modification (requires CONTEXT.md update). **Default answer: production build.**

### 2. Auto-fix LLM choice (D-22 Claude's Discretion)

- **What we know:** D-22 allows "regex patching, AST, LLM-suggested diff".
- **What's unclear:** Whether the executor spawns `claude` (Claude Code CLI) as a subprocess, uses the Anthropic SDK directly, or falls back to simpler regex for common classes of failure.
- **Recommendation:** Three-tier strategy:
  - **Tier 1 (deterministic):** Missing `data-testid="page-root"` → regex-insert on root `<div>` of page.tsx. Zero risk.
  - **Tier 2 (mechanical):** TypeScript/ESLint errors → `ts-morph` AST fix if applicable. Moderate risk.
  - **Tier 3 (LLM):** A11y violations, content assertion failures, CTA wiring → Claude Code subprocess with D-19/D-20 in system prompt + filesystem denylist check on write. Highest risk.
- Planner should pick Tier 1+2 for Plan 02 Task 2 MVP, add Tier 3 only if Tier 1+2 don't converge.

### 3. How to inject fixture session tokens for authenticated probe

- **What we know:** D-10 excludes `/api/*` from page tests; D-13 requires CTA probe.
- **What's unclear:** Some CTAs depend on session cookies being present (e.g., `/posts/[id]/apply` requires a logged-in worker). Setup project (Pattern 2) handles this — but session freshness during a 20-minute full sweep?
- **Recommendation:** Setup project writes `storageState.json`. Each review project loads it at test start. Supabase session tokens are 1-hour JWTs by default — 20-minute sweep is within budget. No explicit refresh needed for v1.1.

### 4. How to simulate the "월말 정산 집계" cron in E2E loop 7 (D-15)

- **What we know:** D-15 loop 7 says "manual via `POST /api/cron/*`".
- **What's unclear:** Existing cron endpoints `cron/auto-approve/route.ts`, `cron/expire-urgent/route.ts` — do they expose a settlement trigger, or is that elsewhere?
- **Recommendation:** Planner inventories `src/app/api/cron/*/route.ts` during Plan 01 Task 2. If a settlement endpoint doesn't exist, Plan 02 loop 7 uses a direct DB operation via the seeded admin client instead. This is within `tests/**` scope per D-19 — no escalation needed.
