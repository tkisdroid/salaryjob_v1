<domain>
## Phase Boundary

**What this phase delivers (scope anchor):**
A self-contained, reproducible QA harness that:

1. **Self-boots** a full Supabase stack locally (Postgres + GoTrue Auth + Storage + Realtime) via `supabase CLI start`.
2. **Self-seeds** the stack with deterministic fixtures (worker/business/admin accounts, jobs, applications, shifts, reviews, settlements) so tests are reproducible across runs.
3. **Self-drives** a real Chromium browser (Playwright) through 54 routes × 2 viewports (Desktop 1280×800 + Mobile 375×812) = **108 route-scenarios**, plus 7 cross-route E2E loops.
4. **Self-verifies** each screen against explicit content assertions (not just "loaded without error") to block white-screen false PASS.
5. **Self-enforces** a full zero-error gate: runtime/console/network/tsc/eslint/a11y critical+serious/performance budget.
6. **Self-fixes** failures via an auto-fix loop with scoped write authority (UI + server actions + API routes); Supabase client layer and Prisma schema core fields are off-limits.
7. **Self-reports** into `07.1-REVIEW.md` with a final boolean `production_ready: true|false`.
8. **Re-runs** until all gates pass — no formal PASS accepted, only real working behavior.

**What this phase does NOT deliver (out of scope — do not add):**
- Real-Supabase production migration apply (→ Phase 8 Day 1, per D-02)
- Visual design system rework, token redesign, component API redesign (→ Phase 9)
- Legacy cleanup, dead-code removal, dependency pruning (→ Phase 10)
- External-service live calls (Toss payments, Kakao Maps key exchange, CLOVA OCR) — these stay MOCKed per `MOCK-INDEX.md`
- Mobile-device farm testing (BrowserStack / real iOS-Android devices) — 375px Chromium emulation only
- Dark mode / RTL / internationalization QA — deferred
- Load testing, k6, stress concurrency — deferred
- CI/CD GitHub Actions wiring — local harness first; CI wiring deferred to a later phase
</domain>

## Implementation Decisions

Each decision is atomic, ID'd, and contains a `why` (rationale) + `apply` (directive) line so downstream agents can act without re-interpretation.

### Category 1 — Local stack bootstrapping

**D-01. Local stack engine = Supabase CLI (`supabase start`, Docker Desktop required).**
- why: full Auth + Storage + RLS + Realtime fidelity; matches production Supabase behavior; avoids mocking auth.users / storage buckets which are central to this app's trust boundary.
- apply: planner must add `supabase` CLI as devDependency or document global install; runner script calls `supabase start` then `supabase status` for health check.

**D-02. Phase 7 Wave 2 (`07-02`, real-Supabase migration apply) is relocated to Phase 8 Day 1.**
- why: local stack takes over the iteration role. Real-Supabase apply becomes a one-shot "production seed" operation better done right before HUMAN-UAT.
- apply: roadmap entry for `07-02` is NOT executed now; its SUMMARY stays absent; STATE.md "Open TODOs" moves the apply step under Phase 8. Planner must NOT treat `07-02` as a blocker for `07.1`.

**D-03. Platform = Windows 11 + WSL2 + Docker Desktop is the documented reference environment.**
- why: primary developer environment.
- apply: `docs/review-harness.md` MUST contain a "Windows/WSL2/Docker Desktop install & verify" section as a first-run checklist; failure paths must output WSL2-friendly error messages.

**D-04. Local stack ports and URLs follow Supabase CLI defaults.**
- apply: `API_URL=http://127.0.0.1:54321`, `DB_URL=postgresql://postgres:postgres@127.0.0.1:54322/postgres`, Studio=`http://127.0.0.1:54323`, Inbucket=`http://127.0.0.1:54324`. These go into `.env.test` (gitignored) written by the bootstrap script.

**D-05. Migration application order for local stack = same as production.**
- apply: runner applies `supabase/migrations/*` in filename ascending order (including `20260414000005_phase6_admin_seed.sql`). Same `scripts/apply-supabase-migrations.ts` is reused with a `--target=local` flag; no new migration runner is written.

### Category 2 — Test data seeding

**D-06. Seed data is deterministic (fixed UUIDs, fixed timestamps via freeze-time).**
- why: snapshot diffing and re-runnability; eliminates flaky tests.
- apply: `scripts/review/seed-test-data.ts` uses hardcoded UUIDs from a `tests/review/fixtures/ids.ts` constants file. Times are set via `vi.setSystemTime` in fixtures and via explicit `created_at` in inserts.

**D-07. Seed coverage = minimum viable full-graph.**
- Worker accounts: 3 (verified / newly-joined / banned)
- Business accounts: 3 (verified / pending-OCR / rejected)
- Admin account: 1 (pre-promoted via seed SQL)
- Jobs: 10 (mix of active / expired / urgent / filled)
- Applications: 5 (mix of pending / accepted / rejected / completed / cancelled)
- Shifts: 2 (scheduled / completed)
- Reviews: 3 (worker→biz / biz→worker / both-sided)
- Settlements: 1 (processed)
- apply: fixtures above are MANDATORY; planner must not reduce the set without updating this decision.

**D-08. Admin account in seed uses a reserved local-only email.**
- apply: seed SQL for local stack substitutes `dev+admin@local.test` as the admin email; this never touches production; does not collide with D-01 of phase `07-02` (which uses a real email).

### Category 3 — Browser-driven verification

**D-09. Browser = Playwright with Chromium only (2 viewport projects: Desktop + Mobile 375px).**
- why: existing `playwright.config.ts` already uses Chromium Desktop; adding a Mobile 375×812 project is a one-liner; Firefox/WebKit deferred to avoid scope creep.
- apply: extend existing `playwright.config.ts` with a second project `{ name: 'mobile-375', use: { ...devices['iPhone 13 Mini'], viewport: { width: 375, height: 812 } } }`. Do NOT create a parallel config file.

**D-10. Route enumeration = auto-discovered from `src/app/**/page.tsx` + manually-curated API routes.**
- apply: `tests/review/routes/discover.ts` globs `src/app/**/page.tsx` and extracts route paths; dynamic segments get placeholder fixtures (from D-07's deterministic IDs). Exclude: `(admin)` routes without admin seed, `/api/*` (those get contract tests, not page tests).

**D-11. Per-route page check = 5 assertions.**
- (a) page status 200 + no redirect loop
- (b) `domcontentloaded` + `networkidle` both fire within 15s
- (c) **explicit content assertion**: at least one `data-testid="page-root"` or route-specific anchor text exists in DOM (per D-14)
- (d) visible text length on body > 50 chars (guards blank/skeleton-stuck states; 50 chosen to filter loading shells like "로딩 중...")
- (e) no loading skeleton / spinner DOM visible after 5s (selector list documented in `tests/review/helpers/loading-states.ts`)
- apply: each assertion is independently FAIL-capable; failure report cites which assertion(s) failed.

**D-12. Per-route a11y = `@axe-core/playwright` with critical + serious levels only.**
- why: moderate/minor produce noise and are borderline style preferences.
- apply: `tests/review/a11y.ts` runs axe on each route-scenario; any critical OR serious violation ⇒ FAIL.

**D-13. Interaction probe (per route, 1 primary CTA).**
- apply: after D-11 assertions, Playwright clicks the route's primary CTA (selector from manifest — see D-14) and asserts ONE of: URL change / modal opened / toast appeared / network request fired. Zero response = FAIL.

**D-14. Per-route manifest file = `tests/review/routes/manifest.ts`.**
- Schema per entry: `{ path: string; desktopOk: boolean; mobileOk: boolean; contentAssertion: string | RegExp; primaryCta: string; seedAs: 'worker' | 'biz' | 'admin' | 'anon' }`.
- why: single source of truth, reviewable, diffable, extendable without touching test code.
- apply: planner creates this manifest as Task 1 of Plan 02 (before any per-route tests). All 54 routes enumerated explicitly even if most share the same CTA shape.

### Category 4 — End-to-end flows (the 7 core loops)

**D-15. Seven canonical E2E loops (each must complete within 60s of simulated clock).**
1. Worker 탐색 → 원탭 지원
2. Business 지원자 확인 → 확정
3. Worker 체크인
4. Worker 체크아웃
5. Worker 리뷰 작성
6. Business 리뷰 작성
7. 월말 정산 집계 (cron trigger manual via `POST /api/cron/*`)
- apply: `tests/review/flows/` has one spec per loop; each spec starts from fresh seed state (seed is re-run between flows).

**D-16. E2E flows are the FOLDED scope for `todo: 2026-04-11-phase-5-plan-05-05-browser-uat-review-settlement-flow.md`.**
- why: the pending todo "Phase 5 Plan 05-05 브라우저 UAT — 리뷰+정산 UI 4개 시나리오" is a subset of loops 5/6/7.
- apply: planner marks this todo as closed when Plan 02 of this phase reaches GREEN. Keep a back-reference in `07.1-SUMMARY.md` so auditors can trace the closure.

### Category 5 — Zero-error gate (FAIL definition)

**D-17. A single run FAILS if ANY of the following is non-zero or out-of-budget:**

| # | Gate | Threshold | Tool |
|---|------|-----------|------|
| G1 | runtime JS errors | 0 | `page.on('pageerror')` |
| G2 | console errors | 0 | `page.on('console')` filter level=error |
| G3 | console **warnings** | 0 | same, level=warning |
| G4 | network 4xx responses (excluding opt-in 401 on auth probes) | 0 | `page.on('response')` |
| G5 | network 5xx responses | 0 | same |
| G6 | TypeScript errors | 0 | `tsc --noEmit` |
| G7 | ESLint errors (warnings allowed) | 0 | `eslint --max-warnings=-1` (warn ok, error not) |
| G8 | axe-core critical + serious | 0 | `@axe-core/playwright` |
| G9 | LCP | ≤ 2.5s | Lighthouse CI |
| G10 | TTI | ≤ 3.5s | Lighthouse CI |
| G11 | initial JS transfer | ≤ 200KB | Lighthouse CI |
| G12 | CLS | ≤ 0.1 | Lighthouse CI |
| G13 | Content assertion per D-11 | all pass | Playwright assertion |
| G14 | CTA interaction probe per D-13 | all pass | Playwright assertion |
| G15 | 7 E2E loops per D-15 | all pass + each ≤ 60s wall-clock | Playwright |
| G16 | existing vitest suite | all pass (0 failures; todos allowed) | `vitest run` |

- apply: the runner aggregates ALL gates before reporting; does not short-circuit on first failure. Full count goes to `07.1-REVIEW.md`.

**D-18. Tolerated exceptions (explicit allow-list, encoded in config not code).**
- `GET /auth/whoami` returning 401 for the anon persona is EXPECTED (not a G4 violation). The allow-list lives in `tests/review/config/allowed-4xx.json`: `[{ route: "/auth/whoami", persona: "anon", status: 401 }]`. Additions require a code comment citing the reason.

### Category 6 — Auto-fix loop

**D-19. Auto-fix writable scope (WHITELIST — deny by default).**
- ✅ `src/components/**`
- ✅ `src/app/**/page.tsx`, `src/app/**/layout.tsx`, `src/app/**/loading.tsx`, `src/app/**/error.tsx`, `src/app/**/not-found.tsx`
- ✅ `src/lib/actions/**`
- ✅ `src/app/api/**/route.ts` (excluding `/api/webhooks/*` — see D-20)
- ✅ `src/lib/services/**` (excluding `src/lib/services/settlement.ts` if it touches payments — see D-20)
- ✅ `src/lib/validations/**`
- ✅ `src/styles/**`, Tailwind classes in any file above
- ✅ `tests/**` (for test-side fixes only; see D-21)

**D-20. Auto-fix DENY list (hard block — fix agent MUST refuse and emit escalation).**
- ❌ `src/lib/supabase/**` (auth/RLS/storage clients — skeleton)
- ❌ `prisma/schema.prisma` core fields: `id`, any `@relation`, `createdAt`/`updatedAt`, required PK/FK. Enum value additions are allowed IF related fix is a user-visible enum (e.g., new job status). Schema structural changes are not.
- ❌ `supabase/migrations/**` (never regenerate migrations mid-review)
- ❌ `.env`, `.env.local`, `.env.example`, `.env.test` (secrets/config perimeter)
- ❌ `next.config.ts`, `package.json` dependencies section (tech-stack lock)
- ❌ `src/app/api/webhooks/**` (signature-verified boundaries — manual review required)
- ❌ `src/lib/services/payment*` if exists (Toss integration boundary)
- ❌ `middleware.ts` (auth/session boundary — one-line fix OK only if the fix agent flags it for human review)
- apply: auto-fix agent receives this DENY list in its system prompt; attempts to edit any path here = immediate abort + `MANUAL-FIX-NEEDED.md` entry.

**D-21. Auto-fix iteration policy.**
- Default max iterations: **3**
- **Progress-aware extension:** if `fail_count[iter_n] < fail_count[iter_n-1]` (strict decrease), allow up to 10 total iterations. If no strict decrease for 3 consecutive iterations → PAUSE and notify user (do NOT auto-stop — await user decision).
- "No strict decrease" is measured by total failure count across all gates G1..G16 aggregated.
- apply: `scripts/review/auto-fix-loop.ts` logs `iter=N failures={prev→curr} progress={yes|no}` per cycle; the decision rule is reproducible.

**D-22. Auto-fix philosophy (overrides formalism).**
- Quote: "실제로 제대로 동작이 가능할 때까지 진행해야 함." (User, 2026-04-15)
- apply: if a page passes all numeric gates but the interaction probe (D-13) or content assertion (D-11) fails, the run FAILS even if G1..G12 report zero. No formal PASS is accepted without functional correctness.

### Claude's Discretion
The following choices are explicitly delegated to the planner/executor — researcher and planner DO NOT need to ask the user:

- Auto-fix fix-generation strategy (regex patching, AST, LLM-suggested diff) — executor's call as long as D-19/D-20 scope is honored
- REVIEW.md layout beyond the mandatory fields listed in the Success Criteria
- Whether to emit junit XML / SARIF / Allure in addition to MD (only MD is required)
</decisions>
